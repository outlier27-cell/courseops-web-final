# UIBE CourseOps UI 参考采集与页面草图

> 目标：先把 Dashboard 和 Kanban 的视觉方向定死，再进入 Figma 高保真。本文档只定义参考、借鉴点、页面结构和验收标准，不进入代码实现。

## 1. 参考来源

### 1.1 Arco Design Pro

参考链接：https://github.com/arco-design/arco-design-pro-vue

用于借鉴：

- Vue3、TypeScript、Pinia、Arco Design 的企业级技术组合。
- 左侧导航 + 顶部栏 + 内容工作区的后台骨架。
- Dashboard、表格、列表、表单、可视化页面的基础组织方式。
- 浅色、克制、清晰的中后台视觉秩序。

不借鉴：

- 默认蓝白模板感。
- 过于通用的指标卡和图表组合。
- 没有课程业务特征的普通后台页面。

CourseOps 改进：

- 用 UIBE 蓝绿色 token 替代默认主色。
- Dashboard 必须体现课程项目、材料提交、风险提醒。
- Kanban 任务卡必须体现课程/项目/材料/截止日期，而不是普通 issue 卡。

### 1.2 Ant Design Pro

参考链接：https://preview.pro.ant.design/

用于借鉴：

- 企业级信息密度。
- 查询表单、表格、筛选、分页、状态标签。
- 可视化页的信息组织方式。
- 数据解释文字与图表结合的方式。

不借鉴：

- React 技术栈本身。
- 太重的通用后台模板风格。
- 过多运营图表堆叠。

CourseOps 改进：

- 项目列表采用 Ant 类信息密度，但视觉使用 Arco + 自定义 token。
- 数据分析页只保留 4 个图表，避免图表 demo 感。
- 图表旁必须有业务解释，例如“风险主要集中在 Web 应用开发课程”。

### 1.3 Linear

参考链接：https://linear.app/

参考链接：https://linear.app/docs/board-layout

用于借鉴：

- 任务流转的清晰状态表达。
- Board/List 视图切换思路。
- 任务卡片的高信息密度和轻量质感。
- 快速、克制、少干扰的交互风格。

不借鉴：

- 深色科技品牌感。
- 纯软件研发 issue 模型。
- 过强的键盘快捷键导向。

CourseOps 改进：

- 任务卡从 issue 改造成课程项目任务。
- 加入材料状态、课程名称、截止日期、风险标签。
- 右侧详情抽屉要服务期末项目演示。

### 1.4 飞书项目 / 飞书 Base Dashboard

参考链接：https://www.feishu.cn/hc/en-US/articles/383808395295-feishu-project-overview

参考链接：https://www.feishu.cn/hc/en-US/articles/161059314076-use-dashboards-in-base

用于借鉴：

- 项目协作场景表达。
- 通过 dashboard 从多个维度分析数据。
- 任务、项目、成员、状态之间的业务关系。
- 工作台里同时展示数据、提醒、动态的方式。

不借鉴：

- 完整协作平台的大范围功能。
- 复杂的组织、成员、权限系统。
- 与本课程期末项目无关的重协作能力。

CourseOps 改进：

- 协作感只体现在负责人、动态、风险提醒，不扩展多人复杂权限。
- Dashboard 聚焦课程项目运营，不做通用企业项目管理。

## 2. 视觉方向锁定

最终方向：

```text
结构像 Arco
信息密度像 Ant
任务体验像 Linear
业务表达像飞书
颜色属于 UIBE CourseOps
```

关键词：

- 克制
- 清晰
- 高级
- 真实系统
- 数字校园
- 课程项目运营

硬性限制：

- 不做 landing page。
- 不做紫色渐变。
- 不做玻璃拟态。
- 不做发光科技风。
- 不做过大标题。
- 不做厚重阴影。
- 不把所有内容都塞进卡片。

## 3. Dashboard 页面草图

### 3.1 信息架构

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Topbar 56px：搜索 / 当前学期 / 用户 / 快捷新建                              │
├──────────────┬───────────────────────────────────────────────────────────────┤
│              │ 页面标题区                                                     │
│ Sidebar      │ UIBE CourseOps 工作台                                          │
│ 232px        │ 课程项目运营总览 · 本周任务风险与材料提交状态                    │
│              │                                                               │
│              │ 指标卡 1       指标卡 2       指标卡 3       指标卡 4           │
│              │ 项目总数       进行中任务     临近截止       本周完成率          │
│              │                                                               │
│              │ ┌─────────────────────────────┐ ┌───────────────────────────┐ │
│              │ │ 项目健康度总览                │ │ 今日风险提醒               │ │
│              │ │ 进度 / 风险 / 材料完成率       │ │ 临期 / 材料 / 高优先级      │ │
│              │ └─────────────────────────────┘ └───────────────────────────┘ │
│              │                                                               │
│              │ ┌───────────────┐ ┌───────────────┐ ┌───────────────────────┐ │
│              │ │ 7天完成趋势    │ │ 课程任务分布    │ │ 最近操作动态            │ │
│              │ └───────────────┘ └───────────────┘ └───────────────────────┘ │
└──────────────┴───────────────────────────────────────────────────────────────┘
```

### 3.2 关键组件

- `PageHeader`：标题、副标题、日期范围、快捷操作。
- `MetricCard`：指标、趋势、解释文案。
- `ProjectHealthPanel`：进度、风险、材料完成率组合。
- `RiskAlertPanel`：今日风险提醒。
- `ChartPanel`：趋势图、柱状图。
- `ActivityTimeline`：操作动态，来自 MongoDB 日志。

### 3.3 视觉规则

- 顶部区域要轻，不做厚重 header。
- 指标卡不靠大阴影，而靠边框、数字层级和状态色。
- 风险提醒要用红/琥珀小面积强调，不整块红色。
- 图表颜色必须使用系统 token。
- 最近动态要像真实日志，不写空泛文案。

### 3.4 示例数据文案

指标卡：

- 项目总数：12，较上周 +2。
- 进行中任务：36，其中高优先级 6。
- 临近截止：8，24 小时内 3。
- 本周完成率：76%，较上周 +12%。

风险提醒：

- Web 应用开发：系统说明书初稿明天截止。
- 数据库实验：ER 图材料未完成。
- 论文写作：高优先级任务 2 天未更新。

最近动态：

- 张同学将“完成系统说明书初稿”从进行中改为待提交。
- 系统记录“Web 期末项目”材料完成率更新为 70%。
- 管理员创建任务“录制 15 分钟系统演示视频”。

### 3.5 Dashboard 验收标准

- 1440 x 900 截图可直接作为说明书主图。
- 第一屏必须同时看到指标、风险、趋势、动态。
- 不出现通用后台 demo 文案。
- 所有模块都能解释为什么和课程项目运营有关。
- 1280 x 720 下不拥挤、不重叠。

```text

```

### 4.5 交互规则

- 点击任务卡打开右侧抽屉。
- 状态切换优先用按钮或下拉，不强制拖拽。
- 状态变化后必须产生操作日志。
- 新建任务使用抽屉或弹窗表单。
- 看板列宽固定，内容多时列内滚动。
- 小屏时看板横向滚动，不挤压卡片。

### 4.6 Kanban 验收标准

- 一眼能看出任务状态、风险和截止日期。
- 任务卡信息密度高但不拥挤。
- 逾期、高优先级、待提交三类状态有明确视觉差异。
- hover 不改变布局尺寸。
- 右侧抽屉打开后主内容不混乱。

## 5. Figma 草图制作顺序

由于当前 Codex 会话没有暴露可调用的 Figma MCP 工具，先按以下顺序准备 Figma 内容。等工具可用或手动打开 Figma 后，直接照此制作。

- Desktop：1440 x 900
- Small Desktop：1280 x 720

### Step 3：先做低保真

只使用灰阶、边框、占位文字，先确认：

- 模块位置。
- 信息密度。
- Dashboard 第一屏是否能放下所有关键信息。
- Kanban 列宽和任务卡密度是否舒服。

### Step 4：再上视觉 token

使用 `courseops-plan.md` 中的 token：

- 主色 `#0F766E`
- 背景 `#F6F8F7`
- 边框 `#E2E8F0`
- 进行中 `#2563EB`
- 待提交 `#F59E0B`
- 完成 `#16A34A`
- 风险 `#DC2626`

### Step 5：选择最终方向

选择标准：

- 哪一版更像真实企业系统。
- 哪一版说明书截图更漂亮。
- 哪一版实现风险更低。
- 哪一版更容易讲清楚课程项目运营业务。

## 6. 推荐最终方向

Dashboard 推荐：`Dashboard B - Command Center`

原因：

- 更适合作为说明书主截图。
- 能同时展示指标、风险、趋势、动态。
- 比普通后台 dashboard 更有课程项目运营特色。

Kanban 推荐：`Kanban A - Linear Inspired`

原因：

- 任务卡质感更高级。
- 信息密度更适合课程任务。
- 状态流转演示更清楚。

最终组合：

```text
Dashboard：飞书式运营工作台 + Arco 布局秩序
Kanban：Linear 式任务卡 + UIBE 课程项目字段
```

由于当前 Figma Starter 计划最多支持 3 个 page，正式文件采用 3 页结构：

已完成内容：

- Strategy 页：明确 Arco、Ant、Linear、飞书分别借鉴什么和不借鉴什么。
- Dashboard A：Dense Enterprise，偏标准企业后台。
- Dashboard B：Command Center，偏运营指挥台。
- Dashboard Final Direction：推荐方向，保留运营指挥台结构，补齐风险队列、项目健康度、动态和趋势洞察。
- Kanban A：Linear Inspired，偏任务卡和右侧详情抽屉。
- Kanban B：Feishu Workspace，偏协作工作台。
- Kanban Final Direction：推荐方向，保留 3 列看板 + 任务详情抽屉，适合 1440 截图完整展示。

当前推荐结论：

- Dashboard 采用 `Dashboard Final Direction - Command Center Refined`。
- Kanban 采用 `Kanban Final Direction - Screenshot Safe Linear`。

第一轮截图检查结论：

- Dashboard 方向正确，适合做说明书主截图；后续高保真阶段需要把图表从占位条升级为更真实的 ECharts 风格图形。
