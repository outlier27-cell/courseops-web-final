# UIBE CourseOps 智慧课程项目运营平台执行计划

> 目标不是做一个功能很多的课程作业，而是做一个第一眼就像真实学校内部系统的高完成度 Web 应用。前端要成为主观加分点，Dashboard 和 Kanban 要能直接放进说明书当主截图。

## 1. 项目定位与交付约束

项目名称：UIBE CourseOps 智慧课程项目运营平台

项目类型：企业级课程项目运营中台

核心目标：围绕课程项目、任务流转、材料提交、风险提醒、数据分析、操作审计，做出一个范围可控但完成度很高的 Web 应用系统。

硬性交付约束来自期末 PDF：

- 独立设计开发一个小功能模块，交互性好，采用企业级开发技术。
- 提交 15 分钟系统视频演示，视频右下角本人出镜。
- 视频中必须打开开发环境介绍代码和技术细节。
- 提交《系统使用说明书》，封面采用学校论文考试封面，内容以界面截图和说明为主。
- 提交学生证或 E 照片，且与视频中本人一致。
- 提交项目全部文件。
- 项目包名必须符合 `cn.edu.uibe.*`，本项目统一使用 `cn.edu.uibe.courseops`。

评分策略：

- 前端第一眼要赢：登录页、Dashboard、Kanban、数据分析/日志页必须精致。
- 技术讲解要稳：Vue 3、Spring Boot、MyBatis-Plus、MySQL、MongoDB、JWT 都有明确可讲位置。
- 演示流程要短而完整：只保证一条主流程绝对稳定，不追求做完整 SaaS。
- 说明书要好看：截图、架构图、流程图、数据库图统一风格。

主流程固定为：

```text
登录
进入 Dashboard
查看风险提醒
创建课程项目
创建任务
在看板中切换任务状态
更新材料清单
查看统计变化
查看 MongoDB 操作日志
```

## 2. 产品范围

### 2.1 P0 必做页面

1. 登录页
2. 工作台 Dashboard
3. 项目列表 / 项目详情
4. 任务看板 Kanban
5. 数据分析 + 操作日志

### 2.2 P1 集成功能

- 材料清单：放在项目详情或任务详情抽屉中，不单独做页面。
- 任务表格视图：如果时间允许，作为看板页辅助视图。
- 通知/风险提醒：在 Dashboard 和任务详情中展示，不单独做复杂通知中心。

### 2.3 P2 默认砍掉

- 系统设置
- 日历视图
- 主题切换
- 真实短信/邮件提醒
- 复杂注册、找回密码、权限矩阵
- 真实文件上传

范围控制原则：

- 不做 10 个页面平均用力。
- Dashboard 和 Kanban 是作品集级页面，优先级最高。
- CRUD 只服务演示主流程。
- MongoDB 只用于操作日志和动态流，降低复杂度。
- 文件上传先做材料记录和状态管理，不强制真实上传。

## 3. 前端设计总方向

视觉关键词：精致、克制、高级、数字校园、运营中台、清晰高效。

前端审美定位：

```text
UIBE CourseOps =
Arco Pro 的企业后台骨架
+ Linear 的任务流转质感
+ 飞书项目的协作工作台表达
+ UIBE 蓝绿色数字校园视觉系统
```

参考收敛：

- 结构主要参考 Arco Design Pro。
- 信息密度和表单/表格规范参考 Ant Design Pro。
- 任务卡片、状态流转、详情抽屉参考 Linear。
- 协作工作台和业务表达参考飞书项目。
- Notion 只参考详情信息组织，不作为主视觉。

必须避免：

- 默认蓝白后台模板感。
- 紫色渐变、玻璃拟态、发光科技风。
- 大标题、大留白、宣传式 landing page。
- 页面铺满悬浮卡片。
- 所有页面都做成同一种卡片堆叠。
- 字号过大、按钮过圆、阴影过重。

前端目标一句话：第一屏就是可操作的真实系统，有数据、有状态、有风险、有动作、有反馈。

## 4. 设计系统硬规范

### 4.1 颜色 Token

基础色：

- 页面背景：`#F6F8F7`
- 次级背景：`#F8FAFC`
- 面板背景：`#FFFFFF`
- 主色：`#0F766E`
- 主色浅背景：`#E6F4F1`
- 强调蓝：`#2563EB`
- 深色文字：`#111827`
- 正文文字：`#334155`
- 次级文字：`#64748B`
- 弱文字：`#94A3B8`
- 边框：`#E2E8F0`
- 分割线：`#EEF2F7`

状态色：

| 语义 | 颜色 | 用法 |
| --- | --- | --- |
| 待开始 | `#64748B` | 灰色标签、弱状态 |
| 进行中 | `#2563EB` | 蓝色标签、进度强调 |
| 待提交 | `#F59E0B` | 琥珀色标签、材料提醒 |
| 已完成 | `#16A34A` | 绿色标签、完成反馈 |
| 已逾期 | `#DC2626` | 红色标签、风险提醒 |
| 高优先级 | `#B91C1C` | 只用于小面积强调 |

颜色使用规则：

- 大面积背景保持浅色和干净，不用强色铺满。
- 状态色只用于标签、图标、局部边框和图表强调。
- 主色 `#0F766E` 用于主按钮、导航选中、关键数字和品牌识别。
- 图表不做单一绿色调，至少使用绿色、蓝色、琥珀、红色、灰色形成业务区分。

### 4.2 字体与字号

推荐字体：

- 中文和正文：系统字体栈，优先保证清晰和稳定。
- 数字：可使用 `DIN` 风格或系统等宽数字特性，突出指标感。

字号规范：

| 用途 | 字号 | 行高 |
| --- | --- | --- |
| 页面标题 | 24px | 32px |
| 区块标题 | 18px | 26px |
| 卡片标题 | 16px | 24px |
| 正文/表格 | 14px | 22px |
| 辅助说明 | 13px | 20px |
| 标签/元信息 | 12px | 18px |
| 指标数字 | 28px / 32px | 36px / 40px |

文字规则：

- 后台工具界面不使用超大 hero 字。
- 字重层级控制在 400 / 500 / 600 / 700。
- 不使用负字距。
- 中文按钮文案必须短，超过 6 个字优先换成图标+tooltip 或放进菜单。

### 4.3 间距、圆角、阴影

间距 Token：

- `4px`：图标与文字、紧凑内部间距
- `8px`：标签组、按钮内部、表格单元
- `12px`：任务卡内部间距
- `16px`：卡片内部常规间距
- `20px`：页面内小区块间距
- `24px`：页面主间距
- `32px`：大区块间距

圆角 Token：

- `6px`：按钮、输入框、标签、筛选器
- `8px`：卡片、表格容器、任务卡、下拉菜单
- `12px`：登录页主面板、抽屉内容块、重要数据面板

阴影 Token：

- 常规面板：不使用明显阴影，靠边框和背景区分。
- 悬浮层/抽屉/弹窗：`0 12px 30px rgba(15, 23, 42, 0.10)`。
- hover：只做轻微边框加深或背景变化，不做夸张浮起。

### 4.4 布局尺寸

桌面主目标：`1440 x 900`

必须兼容：`1280 x 720`、`1440 x 900`、`1920 x 1080`

移动端策略：

- 本项目以桌面端为主。
- 移动端只要求不溢出、不重叠、能纵向浏览。
- Kanban 在小屏允许横向滚动。
- 表格在小屏隐藏低优先级列，保留标题、状态、截止日期、操作。

基础布局：

- 左侧导航：`232px`
- 顶部栏：`56px`
- 内容区 padding：`24px`
- 页面最大内容宽度：不强制居中，后台系统使用全宽工作区。
- 图标尺寸：导航 `18px`，按钮 `16px`，状态图标 `14px`。

## 5. 关键页面设计规格

### 5.1 登录页

定位：第一印象页面，要像企业系统入口，不像普通表单作业。

布局：

- 左侧为品牌与系统价值表达：UIBE CourseOps、课程项目运营、材料提交、风险提醒、审计日志。
- 右侧为登录面板。
- 背景使用浅色校园网格/数据线条纹理，但不能做花哨插画。

组件：

- 账号输入框
- 密码输入框
- 角色切换：学生 / 管理员
- 登录按钮
- 演示账号提示

验收标准：

- 1440 截图第一眼有品牌感。
- 登录按钮、输入 focus、错误提示、加载状态齐全。
- 不做注册和找回密码。

### 5.2 Dashboard 工作台

定位：说明书主截图，老师第一眼判断项目完成度的页面。

桌面布局：

```text
左侧导航 232px
顶部栏 56px
内容区 24px padding

页面头部：
标题 + 副标题 + 日期范围 + 快捷操作

第一行：
4 个指标卡
项目总数 / 进行中任务 / 临近截止 / 本周完成率

第二行：
左 2fr：项目健康度总览
右 1fr：今日风险提醒

第三行：
左 1.2fr：近 7 天任务完成趋势
中 1fr：课程任务分布
右 1fr：最近操作动态
```

必须体现课程项目业务，不做通用“四个统计卡 + 两个图表”模板。

内容要求：

- 指标卡必须有同比/环比或状态解释。
- 风险提醒必须出现“临近截止、材料未完成、高优先级未处理”。
- 最近动态必须能体现 MongoDB 日志来源。
- 项目健康度要结合进度、风险、材料完成率，不只放一个图。

交互：

- 指标卡 hover 显示简短解释。
- 风险提醒点击可跳到任务看板或打开任务详情。
- 图表 tooltip 可用。
- 快捷按钮包括“新建项目”“新建任务”。

验收标准：

- 1440 截图能直接放说明书。
- 1280 宽度不拥挤、不重叠。
- 图表密度清晰，文字不压线。
- 状态色有业务含义。

### 5.3 项目列表 / 项目详情

定位：展示 CRUD、表格筛选、项目进度和材料清单。

项目列表：

- 顶部筛选：课程、状态、风险等级、负责人、关键词。
- 表格列：项目名称、课程、阶段、负责人、截止日期、完成率、风险等级、材料状态、操作。
- 操作：查看详情、编辑、删除。
- 新建项目使用弹窗或抽屉表单。

项目详情：

- 基本信息
- 项目进度
- 关联任务
- 材料提交清单
- 操作动态

材料清单字段：

- 材料类型：说明书、源码、演示视频、截图、学生证照片。
- 状态：未准备、准备中、已完成。
- 完成度：百分比或 checklist。
- 更新时间。

验收标准：

- 表格筛选、分页、创建、编辑可演示。
- 材料清单强绑定期末 PDF 交付要求。
- 项目详情不做成信息堆积，优先按演示流组织。

### 5.4 任务看板 Kanban

定位：第二核心页面，最重要交互页。

看板列：

```text
待开始
进行中
待提交
已完成
已逾期
```

任务卡字段：

```text
任务标题
所属课程 / 项目
优先级
截止日期
负责人
材料状态
风险标签
```

任务卡示例：

```text
完成系统说明书初稿
Web 应用开发 · 期末项目
高优先级 · 6月25日截止
材料：说明书 70%
风险：临近截止
```

看板顶部工具：

- 关键词搜索
- 课程筛选
- 优先级筛选
- 风险筛选
- 新建任务按钮
- 看板 / 表格视图切换，可选

任务详情抽屉：

- 任务基本信息
- 所属项目
- 状态切换
- 材料检查清单
- 操作记录
- 风险说明

交互：

- 点击任务卡打开右侧详情抽屉。
- 状态切换后更新看板列、Dashboard 统计和操作日志。
- 新建/编辑任务后有成功反馈。
- 删除任务需要确认弹窗。
- 如果来不及实现拖拽，可以用状态切换按钮代替；视觉上仍然保持看板质感。

验收标准：

- 看板列宽固定，任务卡高度稳定，hover 不造成布局跳动。
- 任务卡信息密度高但可读。
- 逾期和高优先级能一眼识别。
- 状态变化后日志必须出现新记录。

### 5.5 数据分析 + 操作日志

定位：展示 ECharts、MongoDB 和企业级审计能力。

布局：

- 上半区：4 个核心图表。
- 下半区：操作日志表格或时间线。

保留 4 个图表：

- 近 7 天任务完成趋势：折线图。
- 任务状态占比：环形图。
- 课程任务分布：柱状图。
- 项目健康度：雷达图或健康矩阵。

日志字段：

- 操作时间
- 操作人
- 操作类型
- 业务对象
- 操作摘要
- 来源模块
- MongoDB 日志 ID

筛选：

- 操作类型
- 时间范围
- 关键词

验收标准：

- 图表颜色和系统 token 一致。
- 日志能够说明“为什么用 MongoDB”。
- 操作日志不是静态摆设，主流程动作会生成新日志。

## 6. 组件库清单

Figma 和前端都要沉淀这些组件：

- AppShell：左侧导航 + 顶部栏 + 内容区。
- PageHeader：标题、副标题、操作按钮、日期范围。
- MetricCard：指标卡。
- StatusBadge：状态标签。
- PriorityBadge：优先级标签。
- RiskBadge：风险标签。
- TaskCard：看板任务卡。

- MaterialChecklist：材料清单组件。
- ActivityTimeline：动态/日志时间线。
- FilterBar：筛选工具条。
- DataTable：表格容器。
- ChartPanel：图表面板。
- DetailDrawer：详情抽屉。
- ConfirmDialog：确认弹窗。
- EmptyState：空状态。
- LoadingState：加载状态。
- Toast：成功/失败提示。

组件设计要求：

- 图标优先使用已有图标库，按钮里尽量使用图标表达明确动作。

- 不允许卡片套卡片；需要分组时使用标题、分割线和背景带。
- 所有组件都要有 hover、focus、disabled、loading 或 empty 状态。

## 8. Figma、FigureSpec 与验收工具分工

```text
docs/courseops-ui-reference-and-sketch.md
```

### 8.1 Figma 用途

Figma 只负责 UI 高保真和组件库：

- 创建 UIBE CourseOps 设计文件。
- 建立颜色、字号、间距、圆角 token。
- 设计组件库：按钮、输入框、导航、任务卡、数据卡、筛选器、抽屉、弹窗。
- 输出 4 个关键页面高保真：登录页、Dashboard、任务看板、数据分析/操作日志页。
- Dashboard 和 Kanban 至少做 2 个视觉版本，选最强版本继续细化。
- 前端实现后，用浏览器截图反向校准 Figma。

Figma 不负责正式架构图、流程图、ER 图的最终产物。

### 8.2 FigureSpec 用途

FigureSpec 负责说明书中的正式图件，全部用结构化 JSON 生成 SVG：

- 系统功能结构图。
- 主业务流程图。
- 前后端架构图。
- 数据库关系图。
- MongoDB 操作日志流转图。

输出目录建议：

```text
figures/
  specs/
    system-structure.json
    main-workflow.json
    architecture.json
    database-er.json
    mongo-log-flow.json
  system-structure.svg
  main-workflow.svg
  architecture.svg
  database-er.svg
  mongo-log-flow.svg
```

FigureSpec 图件标准：

- 不嵌套 groups。
- 不画跨 3 行以上的长斜箭头。
- 同类节点字号一致。
- 颜色与前端 token 保持一致。
- SVG 必须可编辑、可复现，说明书中使用 SVG 或导出的高清图片。

每个核心页面检查：

## 12. 风险控制

主要风险：

- 功能太多导致做不完。
- UI 太依赖模板导致不出彩。
- 后端技术点堆太多导致演示不稳定。
- MongoDB、MySQL、JWT 同时做导致调试压力增加。
- 页面都有但每页都不够精致。
- 参考来源太多导致视觉语言不统一。

控制策略：

- 优先完成 P0，P1 只服务主流程，P2 默认砍掉。
- 前端先做静态高保真，保证截图质量。
- Dashboard 和 Kanban 先做到最好看，再补其他页面。
- MongoDB 只做日志，不承载核心业务。
- 登录认证可以简化，但保留企业级代码分层。
- 文件上传先做材料状态，不强制真实上传。
- 如果时间紧，任务中心表格和看板二选一，保留看板。
- 数据分析只保留 4 个核心图表。
